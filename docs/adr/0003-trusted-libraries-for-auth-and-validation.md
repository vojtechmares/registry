# Auth and input validation adopt trusted libraries; everything else stays hand-rolled

The registry was built under a minimal-dependency philosophy, partly justified by the belief that the Cloudflare Workers (workerd) runtime was too constrained to run trusted libraries. That premise no longer holds: workerd shipped a full `node:crypto` in September 2025, and the strongest web-standard auth and validation libraries run on Workers using only WebCrypto and Fetch. Carrying hand-rolled security primitives concentrates the failure modes that are hardest to get right and silent when wrong - `alg: none`, HMAC/RSA algorithm confusion, JWKS key selection, audience/nonce checks - with no external conformance backing, and lets unvalidated OCI manifests and management-API bodies reach deeper layers as loosely-typed `as T` casts.

We relax the philosophy for the security-critical and input-validation layers only, adopting three libraries, each slotting in behind an existing module boundary with no change to any wire format:

- **`jose`** replaces the hand-rolled HS256 in the JWT module (`apps/registry/src/auth/jwt.ts`). Verification pins the accepted algorithm to `HS256`, so the library rejects `alg: none` and algorithm confusion before any crypto runs - the same invariant the hand-rolled code asserts today, now on top of audited base64url/JSON parsing.
- **`oauth4webapi`** (with `jose` for JWKS-backed ID-token verification) replaces the hand-rolled relying-party primitives in `@registry/oidc` - discovery, PKCE, authorization-code exchange, and ID-token verification.
- **Valibot** validates untrusted input: the OCI `ImageManifest`, `ImageIndex`, and `Descriptor` shapes inside `parseManifest`, and the management-API request bodies. It replaces the `as T` casts with a parse that yields the distribution-spec `MANIFEST_INVALID` error or a `400`. Valibot is chosen over Zod on bundle grounds - a representative schema tree-shakes to roughly a tenth of Zod's size, which matters on a bundle-sensitive Worker.

Selection criteria, all required together: web-standard (WebCrypto + Fetch only), zero runtime dependencies, tree-shakeable ESM, explicit Workers support, and conformance backing where it applies (`oauth4webapi` self-certifies the OpenID relying-party Basic and FAPI profiles). Crucially, the chosen libraries need only WebCrypto and Fetch and do **not** require the `nodejs_compat` flag: the runtime-maturity context is what motivates reopening the decision, but nothing adopted here depends on `node:crypto`. These are the first runtime dependencies in the auth, OCI, and core packages, added to each relevant package's manifest.

The relaxation is scoped, not general. What stays hand-rolled, and why:

- **Password hashing** - WebCrypto PBKDF2-HMAC-SHA256 (`apps/registry/src/auth/password.ts`); bcrypt and argon2 are native addons that do not run on workerd.
- **semver** (`@registry/semver`), **cron** (`@registry/cron`), and the **task queue** (`apps/registry/src/tasks/`) - small, self-contained, and no Workers-ready library improves on them.
- **The OCI registry / distribution core** (`@registry/registry-core`) - we play the server role, for which no mature, Workers-ready library exists; the available ones are client-only or unmaintained.
- **Raw D1 access** - stays on raw prepared statements; an ORM or query builder (Drizzle, Kysely) is out of scope and unproven a net win given D1's batch-only atomicity model (no interactive transactions).

Also deferred to their own spikes, and out of scope here: an HTTP routing framework, end-to-end type-safe RPC, PASETO as a token format, and client-side form validation in the dashboard.

The consequences are contained by construction. Each swap is internal: the JWT, OIDC, and manifest modules keep their public interfaces, so callers are untouched. Because the token and OIDC wire formats are unchanged, the deploy keeps in-flight access tokens and login sessions valid - a cross-implementation compatibility test (a token minted by the old code verifies under the new verifier, and the reverse) is the gate that proves it before shipping. Tests pin external accept/reject behaviour rather than library internals, so the suite survives future version bumps. The JWT module's header comment, which today argues against pulling in a JWT library, is updated to state the pinned-algorithm rationale so the code's stated reasoning matches its implementation. The standing cost is three third-party dependencies to track and keep current; the benefit is that the primitives most dangerous to get wrong are now conformance-backed and maintained upstream.
